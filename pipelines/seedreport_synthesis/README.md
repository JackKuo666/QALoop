# 种质报告QA对生成器 (GermplasmReportQAGenerator)

基于大语言模型的种质报告问答对生成器，专为农作物新品种审定报告设计。自动从JSON格式的种质报告生成高质量QA对，用于大模型SFT训练。

## 核心创新点

### 1. 领域专属的8大用户意图体系

针对种质报告设计了覆盖全面的用户意图分类：

| 意图类型 | 示例问题 | 占比要求 |
|---------|---------|---------|
| 微观事实查询 | 品种全生育期多少天？株高多少厘米？ | 10-15% |
| 类型归属确认 | 该品种属于粳稻还是籼稻？ | 10-15% |
| 横向对比分析 | 与对照品种相比增产多少？ | 15-20% |
| 品质评估 | 蛋白质含量如何？适口性怎样？ | 10-15% |
| 栽培操作指南 | 什么时候播种最合适？如何施肥？ | 20-25% |
| 抗病抗虫风险 | 对稻瘟病的抗性如何？ | 5-10% |
| 综合性状综述 | 这个品种的整体表现如何？ | 10-15% |
| 品种溯源追踪 | 品种的亲本来源是什么？ | 5-10% |

### 2. 零指代问答生成

避免使用"该品种"、"本品种"、"它"等指代词，改用品种全名直接提问：

```
❌ 禁止：该品种的全生育期是多少天？
✅ 正确：川引稻2006002的全生育期是多少天？

❌ 禁止：它适合在哪些地区种植？
✅ 正确：郑单958适合在哪些地区种植？
```

### 3. 原子化短回答控制

要求至少40%的问答对为25字以内的原子级短回答，便于SFT训练时模型学习精准的事实性知识。

### 4. 专业口语化表达

| 维度 | 要求 |
|------|------|
| 专业性 | 使用农业领域标准术语（如：全生育期、分蘖力、千粒重） |
| 口语化 | 轻微口语化，避免过于书面的表达 |
| 问句结构 | 50%以品种名开头，50%其他形式（如"全生育期最短的是哪个？"） |

### 5. 严格的内容约束

- **禁止询问**：审定编号、审定年份、审定单位、育种单位等行政元信息
- **禁止询问**：商品名、命名形式、检测机构名称
- **禁止**：存在性问题、"根据报告..."类指向文本来源的表述
- **禁止**：推断性语言（"说明了"、"意味着"）

## 功能特性

### 生成策略
| 类型 | 涵盖意图 | 默认数量 |
|------|---------|---------|
| 问答对 | 8大用户意图（微观事实、类型归属、横向对比、品质评估、栽培指南、抗病抗虫、综合性状、品种溯源） | 8条/报告 |

### 核心特性
- **智能内容过滤**：自动跳过"特征特性"字段为空的报告
- **零指代原则**：避免使用"该品种"、"本品种"等指代表述
- **多维度覆盖**：覆盖8类用户意图中的至少5类
- **专业口语化**：表达偏专业、轻微口语化
- **批量处理**：自动扫描所有物种文件夹，支持按物种组织
- **断点续传**：支持中断恢复，已处理文件自动跳过
- **随机抽样**：支持从每个物种中随机抽取指定数量的文件进行测试

### 性能优化
- 并发数提升至100（可配置）
- 多线程并发处理
- 批次处理机制
- 快速失败与超时控制
- 线程安全的文件写入

## 配置说明

### 环境变量 (.env文件)
```bash
cp .env.example .env
# 编辑 .env 文件，设置 OPENAI_API_KEY

# 可选配置
GERMPLASM_REPORT_PATH=./examples  # 种质报告数据目录
OUTPUT_DIR=QAGenarator            # 输出目录
RECORDS_DIR=Processing_records     # 处理记录目录
```

### 脚本配置参数（在文件顶部集中配置）

| 参数 | 默认值 | 说明 |
|------|-------|------|
| `MAIN_INPUT_PATH` | `./examples` | 种质报告根目录（可通过 GERMPLASM_REPORT_PATH 环境变量覆盖） |
| `OUTPUT_DIR` | `QAGenarator` | 输出目录基础名称（自动添加时间戳） |
| `RECORDS_DIR` | `Processing_records` | 处理记录目录（固定，用于断点续传） |
| `MAX_Q_PER_REPORT` | `8` | 每个报告生成的问答对数量 |
| `SAMPLE_SIZE` | `1000` | 随机抽取文件数量（或"all"处理全部） |
| `MAX_WORKERS` | `100` | 并发线程数 |
| `API_BASE_URL` | `os.getenv("OPENAI_BASE_URL", "https://api.openai.com/v1")` | API地址（通过环境变量） |
| `DEFAULT_MODEL` | `gpt-5.1` | 使用的模型名称 |
| `MAX_OUTPUT_TOKENS` | `8000` | API最大输出tokens |
| `TIMEOUT_SECONDS` | `900` | 单个文件处理超时时间（秒） |
| `BATCH_SLEEP_SECONDS` | `1.0` | 批次间休眠时间（秒） |


```
│   ├── report001.json
│   ├── report002.json
│   └── ...

│   ├── report001.json

│   └── ...
```

### 种质报告文件格式 (JSON)
```json
{
  "品种名称": "川引稻2006002",
  "作物名称": "水稻",
  "特征特性": "[必须字段，为空则跳过]",
  "审定编号": "...",
  "育种单位": "...",
  "产量表现": "...",
  "栽培技术要点": "...",
  "其他字段": "..."
}
```

**重要说明**：
- 每个JSON文件必须包含"特征特性"字段
- 如果"特征特性"字段为空，文件将被跳过并记录
- null值会被自动过滤，不影响处理

## 质量控制

### 问答对生成规则
- **8大用户意图覆盖**：覆盖至少5类意图
  1. 微观事实查询
  2. 类型/性状/归属确认
  3. 横向对比分析
  4. 专业品质评估
  5. 栽培操作指南
  6. 抗病抗虫风险
  7. 综合性状综述
  8. 品种溯源追踪

- **表达要求**：
  - 专业术语规范
  - 轻微口语化
  - 至少40%的QA为25字以内的原子级短回答
  - 问句结构多样化（50%以品种名开头，50%其他形式）

### 严格禁止规则
- **内容禁止**：
  - 询问审定编号、审定年份、审定单位、育种单位等行政元信息
  - 询问作物的商品名或命名形式
  - 询问检测、鉴定、试验机构名称
  - 存在性问题或指向"报告本身"的问题

- **表述禁止**：
  - 使用"该品种"、"本品种"、"它"等指代表述
  - 使用"根据报告"、"报告中显示"、"审定数据中"等指向文本来源的表述
  - 推断性语言：如"说明了"、"意味着"、"由此可见"
  - 教科书式因果解释或长铺垫说明

### 长度验证
- 问题≥8字符
- 答案≥10字符

## 断点续传

### 工作原理
1. 处理记录保存在固定的 `Processing_records/` 目录
2. 每次运行前自动加载已处理文件列表
3. 已处理文件自动跳过
4. 中断后可继续执行，不会重复处理

### 记录文件说明
- **追加模式**：所有处理记录采用追加模式，保留历史记录
- **线程安全**：多线程并发写入时使用锁保护
- **固定目录**：记录目录不带时间戳，永久保存

### 清除处理记录
如需重新处理所有文件，删除相应的记录文件：
```bash
# 删除某个物种的处理记录
rm Processing_records/maize_processed_records.jsonl
rm Processing_records/maize_skipped_records.jsonl

# 或删除所有处理记录
rm -rf Processing_records/
```

## 性能优化

### 并发处理
- **线程池**：使用 `ThreadPoolExecutor` 实现多线程并发
- **批次处理**：分批提交任务，避免一次性提交过多任务
- **批次休眠**：批次之间休眠，避免API限流

### 超时控制
- **单文件超时**：默认900秒（15分钟）
- **慢速文件警告**：超过60秒会输出警告

### 线程安全
- **文件写入锁**：每个输出文件对应独立的锁
- **统计信息锁**：使用锁保护共享统计变量

## 常见问题

| 问题 | 解决方案 |
|------|---------|
| API调用失败 | 检查API密钥和网络连接，查看错误信息 |
| 生成QA为空 | 检查报告内容是否充足，"特征特性"字段是否为空 |
| 文件被跳过 | 查看 `{物种名}_skipped_records.jsonl` 了解跳过原因 |
| 处理速度慢 | 适当增加 `MAX_WORKERS` 并发数（注意API限流） |
| 成本过高 | 减小 `SAMPLE_SIZE` 先测试，确认无误后再处理全部 |
| 断点续传失败 | 检查 `Processing_records/` 目录是否存在且可写 |
| 输出目录覆盖 | 每次运行自动添加时间戳，不会覆盖之前的结果 |

## 注意事项

1. **输入目录结构**：确保输入目录结构正确，每个物种有独立的文件夹
2. **测试先行**：建议先用少量文件测试（设置 `SAMPLE_SIZE`），确认无误后再处理全部
3. **时间戳目录**：每次运行创建新的带时间戳的输出目录，不覆盖之前的QA数据
4. **追加模式**：处理记录采用追加模式，保留所有历史处理信息
5. **特征特性字段**：必须包含且非空，否则文件将被跳过
6. **并发配置**：根据API限制和服务器性能合理设置并发数
7. **成本控制**：大规模处理前先用小样本估算成本
